## 1. Постановка сценария

Рассмотрим типичный сценарий взаимодействия клиент-сервер в платёжной системе. Клиент (мобильное приложение, браузер, сторонний сервис) отправляет POST-запрос на оплату заказа. Запрос успешно достигает сервера, сервер начинает обработку, списывает средства и формирует ответ. Однако в момент передачи ответа клиенту происходит обрыв сетевого соединения: TCP-соединение разрывается, клиент получает таймаут или ошибку сети. С точки зрения клиента операция не завершилась, и он не знает, была ли оплата выполнена. Клиент принимает решение повторить запрос, используя тот же самый payload (идентификатор заказа, сумму и т.д.).

Без защиты от повторных запросов на уровне API второй запрос снова вызовет бизнес-логику списания. Если заказ к этому моменту уже оплачен (первый запрос успел выполниться), сервер вернёт ошибку "already paid", что корректно, но создаёт лишнюю нагрузку и путаницу. Хуже ситуация, когда первый запрос ещё находится в обработке и два запроса выполняются параллельно: в этом случае возможно двойное списание (race condition), что было продемонстрировано в ЛР2.

Идемпотентность платежных запросов решает эту проблему следующим образом: клиент при первом запросе генерирует уникальный ключ (Idempotency-Key) и передаёт его в заголовке. Если запрос нужно повторить из-за сбоя сети, клиент отправляет тот же ключ. Сервер при получении запроса проверяет: если запрос с таким ключом и таким же телом уже был обработан, сервер возвращает сохранённый (кэшированный) ответ без повторного выполнения бизнес-логики. Это гарантирует, что операция списания произойдёт ровно один раз, независимо от количества сетевых повторов.

## 2. Реализация таблицы idempotency_keys

Таблица `idempotency_keys` спроектирована для хранения всей необходимой информации об обработанных идемпотентных запросах. Ниже приведён итоговый DDL с пояснениями каждой колонки:

```sql
CREATE TABLE IF NOT EXISTS idempotency_keys (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    idempotency_key VARCHAR(255) NOT NULL,
    request_method VARCHAR(16) NOT NULL,
    request_path TEXT NOT NULL,
    request_hash TEXT NOT NULL,
    status VARCHAR(32) NOT NULL DEFAULT 'processing',
    status_code INTEGER,
    response_body JSONB,
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    expires_at TIMESTAMP WITH TIME ZONE NOT NULL,
    CONSTRAINT idempotency_status_check CHECK (status IN ('processing', 'completed', 'failed'))
);
```

**Назначение полей:**

- `id` (UUID) — уникальный идентификатор записи. Используется `uuid_generate_v4()` для автоматической генерации.
- `idempotency_key` (VARCHAR 255) — ключ, передаваемый клиентом в заголовке `Idempotency-Key`. Это строка, сгенерированная клиентом (например, UUID v4). Именно по этому полю сервер идентифицирует повторные запросы.
- `request_method` (VARCHAR 16) — HTTP-метод запроса (POST, PUT, PATCH). В рамках данной работы проверяются только POST-запросы к платёжным endpoint'ам.
- `request_path` (TEXT) — путь endpoint'а (например, `/api/payments/retry-demo`). Позволяет использовать один и тот же `Idempotency-Key` для разных endpoint'ов.
- `request_hash` (TEXT) — SHA-256 хэш тела запроса. Критически важное поле: оно позволяет обнаружить ситуацию, когда клиент повторно использует один и тот же `Idempotency-Key` с другим payload (другой заказ, другая сумма). В этом случае сервер возвращает 409 Conflict вместо повторного выполнения операции.
- `status` (VARCHAR 32) — статус обработки запроса. Может принимать три значения: `processing` (запрос выполняется), `completed` (запрос успешно обработан и результат закэширован), `failed` (запрос завершился ошибкой, результат также сохраняется). CHECK-ограничение гарантирует, что в поле не могут попасть произвольные значения.
- `status_code` (INTEGER) — HTTP статус-код оригинального ответа. Сохраняется для точного воспроизведения кэшированного ответа.
- `response_body` (JSONB) — тело ответа сервера, сохранённое в формате JSON. JSONB выбран для возможности произвольной структуры ответа и быстрого поиска по полям JSON. Именно это поле позволяет middleware вернуть клиенту тот же ответ без обращения к бизнес-логике.
- `created_at`, `updated_at`, `expires_at` (TIMESTAMP WITH TIME ZONE) — временные метки создания, обновления и истечения срока действия записи. `expires_at` используется для периодической очистки устаревших записей.

**Ограничения и индексы:**

- `UNIQUE (idempotency_key, request_method, request_path)` — уникальный индекс, гарантирующий, что одна пара (ключ, endpoint) может иметь только одну запись. Это основа защиты: при попытке вставить дубликат конкурентным запросом используется `ON CONFLICT DO NOTHING`.
- `idx_idempotency_lookup` — составной индекс по (idempotency_key, request_method, request_path, status) для быстрого поиска записи при каждом запросе middleware.
- `idx_idempotency_expires` — индекс по `expires_at` для эффективной очистки просроченных записей (например, через cron-задачу `DELETE FROM idempotency_keys WHERE expires_at < NOW()`).
- Триггер `trg_idempotency_updated_at` автоматически обновляет поле `updated_at` при каждой модификации записи.

## 3. Реализация middleware

Алгоритм middleware `IdempotencyMiddleware` реализован в файле `backend/app/middleware/idempotency_middleware.py` и работает по следующей пошаговой логике:

**Шаг 1: Фильтрация запросов.** Middleware проверяет, является ли запрос POST-методом к одному из путей в белом списке (`/api/payments/retry-demo`, `/api/payments/pay`). Если нет — запрос пропускается без изменений через `call_next(request)`. Это гарантирует, что middleware не влияет на GET, PUT, DELETE запросы и другие endpoint'ы.

**Шаг 2: Чтение Idempotency-Key.** Из заголовков запроса извлекается значение `Idempotency-Key`. Если заголовок отсутствует, middleware пропускает запрос без обработки идемпотентности. Это обеспечивает обратную совместимость с клиентами, которые ещё не поддерживают Idempotency-Key.

**Шаг 3: Хэширование тела запроса.** Тело запроса (raw bytes) хэшируется алгоритмом SHA-256. Полученный хэш используется для сверки с сохранённым хэшем при повторном запросе. Это позволяет обнаружить reuse ключа с другим payload.

**Шаг 4: Поиск в БД.** В рамках транзакции выполняется SELECT из таблицы `idempotency_keys` по комбинации (idempotency_key, request_method, request_path). Возможны три сценария:

- **4a: Запись найдена, статус `completed`, хэш совпадает.** Это означает, что запрос с таким ключом и payload уже был обработан. Middleware формирует JSONResponse с сохранённым `response_body` и `status_code`, добавляет заголовок `X-Idempotency: cached` и немедленно возвращает его клиенту. Бизнес-логика оплаты не вызывается.
- **4b: Запись найдена, но хэш отличается.** Это означает попытку reuse одного `Idempotency-Key` с другим payload. Middleware возвращает 409 Conflict с детальным сообщением об ошибке. Это предотвращает ситуацию, когда клиент по ошибке использует один и тот же ключ для разных операций.
- **4c: Запись не найдена.** Middleware вставляет новую запись со статусом `processing`. Используется `ON CONFLICT DO NOTHING` для обработки конкурентных запросов с одним и тем же ключом: если два запроса пришли одновременно, только один создаст запись, а второй увидит `processing` и пропустит проверку.

**Шаг 5: Выполнение downstream-запроса.** Вызывается `call_next(request)` для передачи запроса к обработчику (business logic). Response body читается полностью для кэширования.

**Шаг 6: Сохранение результата.** После получения ответа middleware обновляет запись в `idempotency_keys`: устанавливает статус `completed` (или `failed` при 5xx ошибках), сохраняет `status_code` и `response_body` в формате JSONB.

**Шаг 7: Возврат ответа.** Оригинальный response возвращается клиенту.

## 4. Демонстрация без защиты

Для демонстрации проблемы повторной оплаты без идемпотентности реализован тест `test_retry_without_idempotency_can_double_pay` в файле `backend/app/tests/test_retry_without_idempotency.py`.

**Методика проведения:** Создаётся тестовый заказ в статусе `created`. Затем запускаются две параллельные попытки оплаты через `PaymentService.pay_order_unsafe()`, который использует уровень изоляции READ COMMITTED без блокировок FOR UPDATE. Перед проверкой статуса в обеих транзакциях отключаются триггеры защиты от двойной оплаты (через `SET LOCAL app.bypass_payment_check = 'true'`), что имитирует ситуацию без триггеров на уровне БД.

**Результат:** В таблице `order_status_history` появляются две записи со статусом `paid` для одного и того же заказа. Это подтверждает, что без идемпотентности повторный запрос приводит к двойному списанию. Оба конкурентных запроса успешно прошли проверку статуса (оба увидели `status = 'created'`), и оба выполнили UPDATE до `'paid'`, каждый вставив запись в историю.

**Запуск теста:**
```bash
docker compose exec -T backend pytest app/tests/test_retry_without_idempotency.py -v -s
```

Тест проходит успешно, подтверждая наличие проблемы: количество paid-событий в истории равно 2 или больше, что является недопустимым для платёжной системы.

## 5. Демонстрация с Idempotency-Key

Для демонстрации защиты с идемпотентностью реализован тест `test_retry_with_same_key_returns_cached_response` в файле `backend/app/tests/test_retry_with_idempotency_key.py`.

**Методика проведения:** Создаётся тестовый заказ в статусе `created`. Первый POST-запрос к `/api/payments/retry-demo` отправляется с заголовком `Idempotency-Key: test-key-retry-123` и payload `{"order_id": "...", "mode": "for_update"}`. После получения успешного ответа отправляется повторный запрос с тем же ключом и тем же payload.

**Результат первого запроса:** Статус-код 200, тело ответа содержит `"success": true`, заказ переведён в статус `paid`.

**Результат повторного запроса:** Статус-код 200, тело ответа идентично первому запросу. В заголовках ответа присутствует `X-Idempotency: cached`, что однозначно указывает на возврат кэшированного ответа из таблицы `idempotency_keys`. Проверка истории заказа показывает только одно событие `paid` — повторного списания не произошло.

**Дополнительная проверка через SQL:**
```sql
SELECT * FROM idempotency_keys ORDER BY created_at DESC LIMIT 10;
```
Подтверждает, что запись с данным ключом существует, статус `completed`, и `response_body` содержит сохранённый JSON ответа.

**Запуск теста:**
```bash
docker compose exec -T backend pytest app/tests/test_retry_with_idempotency_key.py -v -s
```

## 6. Негативный сценарий

Для проверки обработки некорректного reuse ключа реализован тест `test_same_key_different_payload_returns_conflict` в том же файле.

**Методика:** Первый запрос отправляется с `Idempotency-Key: test-key-conflict-456` и payload `{"order_id": "...", "mode": "for_update"}`. Второй запрос отправляется с тем же ключом, но другим payload `{"order_id": "...", "mode": "unsafe"}`. Изменение `mode` приводит к другому SHA-256 хэшу тела запроса.

**Результат:** Второй запрос получает статус-код 409 (Conflict) с телом:
```json
{
  "detail": "Idempotency-Key 'test-key-conflict-456' was already used with a different request payload for POST /api/payments/retry-demo"
}
```

Это корректное поведение: middleware обнаружил, что хэш нового payload не совпадает с сохранённым, и заблокировал операцию, вернув информативную ошибку клиенту. Это защищает от случайной или намеренной подмены payload при повторе запроса.

## 7. Сравнение с решением из ЛР2 (FOR UPDATE)

Для сравнительного анализа реализован тест `test_compare_for_update_and_idempotency_behaviour` в файле `backend/app/tests/test_compare_idempotency_vs_for_update.py`. Результаты теста позволяют чётко разграничить два подхода.

**FOR UPDATE (решение из ЛР2):**

1. **Цель механизма:** Предотвратить race condition при конкурентных транзакциях. Гарантировать, что только одна транзакция может прочитать и изменить статус заказа, блокируя строку на уровне БД.
2. **Что происходит при повторе запроса:** Вторая параллельная транзакция блокируется на `SELECT ... FOR UPDATE` до завершения первой. После коммита первой транзакции вторая видит `status = 'paid'` и получает бизнес-ошибку `OrderAlreadyPaidError`. То есть вторая попытка получает **ошибку**.
3. **Уровень гарантии:** БД-слой (PostgreSQL). Транзакционная изоляция REPEATABLE READ +悲观ная блокировка строки.
4. **UX для клиента:** Клиент получает ошибку при повторе. Он не знает, был ли первый запрос успешен — возможна путаница.

**Idempotency-Key + middleware (ЛР4):**

1. **Цель механизма:** Гарантировать, что повторный запрос клиента (например, после обрыва сети) не приведёт к повторному выполнению операции и вернёт тот же результат. Защита на уровне API-контракта.
2. **Что происходит при повторе запроса:** Второй запрос с тем же ключом и payload получает **тот же успешный ответ** из кэша (middleware возвращает сохранённый JSON без вызова бизнес-логики). Ошибки нет.
3. **Уровень гарантии:** API-слой (middleware + таблица idempotency_keys в БД).
4. **UX для клиента:** Клиент получает тот же успешный ответ. Семантически это означает, что операция выполнена ровно один раз, и клиент может доверять результату.

**Могут ли оба механизма использоваться вместе?** Да, и в реальных платёжных системах они дополняют друг друга. FOR UPDATE защищает от конкурентных транзакций на уровне БД (race condition между двумя разными клиентами или сервисами). Idempotency-Key защищает от повторных запросов одного и того же клиента на уровне API (network retry). Использование обоих механизмов вместе обеспечивает двухуровневую защиту: конкурентные запросы разных клиентов блокируются FOR UPDATE, а повторные запросы одного клиента (с тем же Idempotency-Key) обрабатываются из кэша middleware.

## 8. Выводы

1. **Идемпотентность — критически важное требование для платёжных API.** В распределённых системах с сетевыми задержками, таймаутами и обрывами соединения повтор запроса — это не исключение, а норма. Без идемпотентности на уровне API даже корректная клиентская логика может привести к двойному списанию.

2. **Idempotency-Key и FOR UPDATE решают разные задачи и не взаимозаменяемы.** FOR UPDATE защищает от конкурентных транзакций на уровне строк БД, обеспечивая целостность данных при параллельном доступе. Idempotency-Key защищает от повторных запросов клиента на уровне API, обеспечивая предсказуемый результат при сетевых сбоях. В продакшен-системах рекомендуется использовать оба механизма вместе.

3. **Хэширование тела запроса — необходимый механизм обнаружения некорректного reuse ключа.** Сохранение SHA-256 хэша payload позволяет серверу обнаружить ситуацию, когда клиент повторно использует один и тот же Idempotency-Key с другим телом запроса. Без этой проверки возможна ситуация, когда кэшированный ответ от предыдущей операции будет некорректно возвращён для совершенно другой операции.

4. **Кэширование ответов в JSONB обеспечивает эффективность и гибкость.** Сохранение полного тела ответа (status_code + JSON body) в таблице `idempotency_keys` позволяет middleware вернуть точную копию оригинального ответа без обращения к бизнес-логике. JSONB обеспечивает гибкость для произвольных структур ответов и возможность будущей аналитики по кэшированным операциям.

5. **Middleware-подход инкапсулирует логику идемпотентности, не загрязняя бизнес-код.** Реализация идемпотентности на уровне middleware позволяет добавить защиту ко всем целевым endpoint'ам без модификации сервисного слоя или API-роутов. Это соответствует принципу separation of concerns и упрощает тестирование и поддержку кода.
